----------------------------------------------------------------
A. INFORMASI PROYEK
----------------------------------------------------------------
Nama Proyek   : Caching Strategy Simulator
Deskripsi     : Simulasi strategi caching menggunakan in-memory
                cache (mensimulasikan Redis/Memcached) di depan
                database SQLite. Mengimplementasikan cache-aside
                pattern dengan berbagai eviction policy dan
                invalidation strategy.

Stack Teknologi:
  - Backend    : Node.js + Express.js
  - Database   : SQLite (via better-sqlite3)
  - Cache      : In-memory (CacheEngine custom)
  - Frontend   : HTML + CSS + Vanilla JS
  - Port       : 3000

----------------------------------------------------------------
B. FITUR YANG DIUJI
----------------------------------------------------------------
1. Cache Engine     — LRU, LFU, FIFO, TTL eviction policy
2. Cache-Aside      — GET product/user/order dari DB + cache
3. Write Strategies — Write-through, Write-back, Write-around
4. CDN Simulator    — Edge cache 4 region, purge mechanism
5. Invalidation     — TTL sweep, Tag-based, Key-based, Write-through refresh

SKENARIO 1 — CACHE MISS DAN CACHE HIT (Cache-Aside Pattern)
════════════════════════════════════════════════════════════════

Tujuan:
  Membuktikan bahwa request pertama (MISS) membutuhkan waktu
  lebih lama karena query ke DB, sedangkan request berikutnya
  (HIT) jauh lebih cepat karena diambil dari cache.

Langkah Pengujian:
  1. Buka tab "DB Queries"
  2. Pilih tipe data = Product, ID = 1
  3. Klik tombol [Query]
  4. Catat: Source dan Latency pada hasil pertama
  5. Klik tombol [Query] sekali lagi (tanpa flush cache)
  6. Catat: Source dan Latency pada hasil kedua

Hasil yang Diharapkan:
  Request 1 : Source = DB   | Latency = 50–150ms (simulasi I/O)
  Request 2 : Source = CACHE | Latency = 0–5ms

  Request 2 : Source = ________ | Latency = ________

SKENARIO 2 — PERBANDINGAN EVICTION POLICY
════════════════════════════════════════════════════════════════

Tujuan:
  Membandingkan hit rate antara LRU, LFU, FIFO, dan TTL
  menggunakan pola akses yang sama.

Pola akses yang digunakan (urutan query):
  product:1 → product:2 → product:3 → product:1 →
  product:4 → product:1 → product:2 → product:5 →
  product:1 → product:3

Langkah Pengujian:
  1. Buka tab "Cache Engine"
  2. Set Max Size = 4 slots, lalu klik [Apply]
  3. Ubah policy ke LRU, klik [Apply]
  4. Jalankan urutan query di atas via tab "DB Queries"
  5. Catat Hit Rate dan Evictions di tab "Cache Engine"
  6. Klik [Flush cache], ganti policy ke LFU, ulangi langkah 4-5
  7. Ulangi untuk FIFO dan TTL

----------------------------------------------------------------
D. KESIMPULAN PENGUJIAN
----------------------------------------------------------------

1. Cache-Aside Pattern:
   Terbukti mengurangi beban DB secara signifikan. Setelah data
   masuk cache, latency turun dari ~50–150ms (DB) menjadi <5ms.

2. Eviction Policy Terbaik:
   - LRU cocok untuk workload dengan temporal locality tinggi
   - LFU cocok untuk konten populer yang sering diakses
   - TTL cocok untuk data yang punya masa berlaku alami

3. Write Strategy Trade-off:
   - Write-through  : konsistensi tinggi, latency lebih tinggi
   - Write-back     : performa tinggi, risiko data loss
   - Write-around   : cocok untuk data yang jarang dibaca ulang

4. CDN:
   Origin fetch (MISS) 10–20x lebih lambat dibanding edge hit.
   Resource statis (gambar, CSS) sangat diuntungkan CDN.
   Resource dinamis (API) butuh TTL pendek untuk freshness.

5. Invalidation:
   Tag-based paling fleksibel untuk invalidasi massal.
   Key-based paling presisi untuk satu data spesifik.
   TTL sweep cocok untuk pembersihan rutin otomatis.
